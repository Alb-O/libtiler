Below is a concrete v1 spec. I would treat this as the normative design unless you decide to widen scope later.

It is intentionally narrower than a WM tiler:

- single root tree
- binary splits only
- all leaves always present in the solved layout
- no gaps, borders, tabs, stacks-as-special-nodes, floating, or multi-root workspaces
- exact integer geometry in arbitrary units
- pure deterministic solver
- thin state machine owning focus and single selection

That narrower scope buys you a much cleaner mental model, stronger invariants, and much harder-to-fake tests.

# 1. Design summary

The library is two layers.

**Core layer**

- owns the split tree
- owns leaf metadata: hard size limits and priorities
- exposes pure structural operations
- exposes a pure solver: `Tree + Bounds + Policy -> Snapshot`

**State layer**

- owns `focus` and `selection`
- exposes geometry-dependent commands like directional focus and resize
- requires a verified `Snapshot` for geometry-dependent commands
- never stores external root bounds internally

The solver is **best-effort but certifying**:

- it always returns a full partition of the root rectangle
- it also reports whether all hard constraints were satisfied
- strict callers can reject any snapshot with violations

That is the right fit for a TUI/editor split system, because the terminal can always shrink below your preferred minimums.

# 2. Goals and non-goals

## Goals

- deterministic layout for a single split tree

- stable leaf identity across ordinary edits

- exact structural semantics

- clear distinction between:

  - topology
  - preferred sizing
  - hard limits
  - focus/selection
  - solved geometry

- support for:

  - insert/split/remove/move/swap
  - subtree transforms
  - geometric navigation
  - multiple resize strategies
  - default layout presets
  - hard min/max constraints
  - diagnostic reports

- strong property and oracle testing

## Non-goals for v1

- multi-root/workspace/monitor support
- gaps, borders, padding
- tabs or stacked hidden leaves
- “monocle” as a core node type
- aspect-ratio constraints
- cell-specific logic
- n-ary containers

`Monocle` is deliberately out of the normative core. It breaks the strongest invariant in this design: **every solve returns an exact partition of the full root rectangle among visible leaves**. If you want monocle later, treat it as a projection/view mode layered above the split solver.

# 3. Mental model

The key model is:

**The tree stores preferences, not pixels.**

A split node says:

- which axis it divides on
- which subtree is leading vs trailing
- what relative share each side prefers

A leaf says:

- who it is
- how small it is allowed to get
- optionally how large it prefers not to exceed
- how important it is under shortage pressure

The solver takes those preferences and a root rectangle and projects them into exact rectangles.

That means:

- operations mutate **structure and preferences**
- solving computes **derived geometry**
- solve never mutates the tree
- geometric commands consume a `Snapshot`, because navigation and directional resize depend on current solved rectangles

This separation is extremely important.

# 4. Core semantic model

## 4.1 Geometry

Use half-open rectangles:

```rust
pub struct Rect {
    pub x: i32,
    pub y: i32,
    pub w: u32,
    pub h: u32,
}
```

Semantics:

- covered x-range is `[x, x + w)`
- covered y-range is `[y, y + h)`

Why:

- exact partitioning is simpler
- adjacency is non-overlap by construction
- rasterized proof tests are easy

Use arbitrary integer units. In a TUI, those units are cells. In another client, they can be pixels or logical points.

## 4.2 Axes and slots

Use axis names that cannot be misread:

```rust
pub enum Axis {
    X, // left/right split, consumes width
    Y, // top/bottom split, consumes height
}

pub enum Slot {
    A, // leading: left or top
    B, // trailing: right or bottom
}

pub enum Direction {
    Left,
    Right,
    Up,
    Down,
}
```

For `Axis::X`:

- `A` is left
- `B` is right

For `Axis::Y`:

- `A` is top
- `B` is bottom

## 4.3 Node kinds

```rust
pub enum Node<T> {
    Leaf(LeafNode<T>),
    Split(SplitNode),
}
```

Leaves hold user payload plus metadata. Splits hold topology and preferred share.

## 4.4 Identity

Use opaque `NodeId`. Recommend monotonically increasing `u64` IDs, never reused within a session.

That gives:

- simple replay
- stable debugging
- fewer “generation” headaches
- better snapshot diffs

Recommendation:

- leaf IDs are semantically stable as long as the leaf survives
- split IDs are stable until that split is removed or a subtree is rebuilt

The external world should depend primarily on leaf IDs, not split IDs.

## 4.5 Weights vs priorities vs limits

These are separate.

### Weights

Preferred share between siblings along the split axis.

```rust
pub struct WeightPair {
    pub a: u32,
    pub b: u32,
}
```

Semantics:

- only relative proportion matters
- `(3, 2)` means A ideally wants 60% and B 40%
- zero is allowed on one side
- `(0, 0)` is invalid

Allowing zero is useful. It lets an interactive resize express “prefer collapsed” without inventing special cases.

### Priorities

How painful it is to short this leaf under infeasible shortage.

```rust
pub struct Priority {
    pub shrink: u16, // higher => protect this leaf more under shortage
    pub grow: u16,   // reserved hook; v1 core mostly uses shrink
}
```

Default:

```rust
Priority { shrink: 1, grow: 1 }
```

### Limits

Hard authored size bounds on leaves.

```rust
pub struct SizeLimits {
    pub min_w: u32,
    pub min_h: u32,
    pub max_w: Option<u32>,
    pub max_h: Option<u32>,
}
```

Derived subtree limits are computed bottom-up.

# 6. Tree invariants

These should be explicitly validated by a `validate()` routine.

## Structural invariants

If `root.is_none()`, the tree is empty.

Otherwise:

- root exists in `nodes`
- root has `parent == None`
- every non-root node has exactly one parent
- every split has exactly two distinct children
- every child points back to the split as parent
- no cycles
- all nodes reachable from root
- no unreachable nodes
- no split with a missing child
- no split with identical child IDs

## Metadata invariants

- `weights.a + weights.b > 0`

- for each leaf:

  - if `max_w` exists: `min_w <= max_w`
  - if `max_h` exists: `min_h <= max_h`

- priorities must be nonzero or normalized to at least 1

## Normal form

After deletion:

- any split that would become unary must be collapsed immediately
- the remaining child is spliced upward
- the old split node is removed

That keeps the topology canonical and simplifies proofs.

# 7. Solver model

The solver is pure:

```rust
pub fn solve<T>(
    tree: &Tree<T>,
    root: Rect,
    policy: &SolverPolicy,
) -> Snapshot
```

It returns:

- rectangles for all nodes
- a leaf placement map
- split allocation traces
- violation diagnostics
- a `strict_feasible` flag

## 7.1 Subtree summary

This is computed bottom-up and is exact for v1 because constraints are axis-separable.

```rust
pub struct Summary {
    pub min_w: u32,
    pub min_h: u32,
    pub max_w: Option<u32>,
    pub max_h: Option<u32>,
    pub leaf_count: u32,
    pub shrink_cost: u64,
    pub grow_cost: u64,
}
```

For a leaf:

- `min_*` and `max_*` come from leaf limits
- `leaf_count = 1`
- `shrink_cost = priority.shrink as u64`
- `grow_cost = priority.grow as u64`

For a split on `Axis::X`:

- `min_w = a.min_w + b.min_w`
- `min_h = max(a.min_h, b.min_h)`
- `max_w = a.max_w + b.max_w` if both finite, else `None`
- `max_h = min_finite(a.max_h, b.max_h)`
- `leaf_count = a.leaf_count + b.leaf_count`
- `shrink_cost = a.shrink_cost + b.shrink_cost`
- `grow_cost = a.grow_cost + b.grow_cost`

For a split on `Axis::Y`, swap width/height roles:

- `min_w = max(a.min_w, b.min_w)`
- `min_h = a.min_h + b.min_h`
- `max_w = min_finite(a.max_w, b.max_w)`
- `max_h = a.max_h + b.max_h` if both finite, else `None`

This exactness depends on **not** supporting aspect-ratio constraints in v1.

## 7.2 Allocation at a split

This is the normative heart of the solver.

Suppose a split on axis `α` receives extent `E` along that axis. Let `a` be the extent assigned to child `A`. Then child `B` gets `E - a`.

Define:

- `min_a`, `min_b`: subtree mins on axis `α`
- `max_a`, `max_b`: subtree maxes on axis `α`
- `wa`, `wb`: split weights
- `sa`, `sb`: shortage penalty weights, usually subtree `shrink_cost`

For every integer `a` in `0..=E`, define:

```text
size_a = a
size_b = E - a

short_a = max(0, min_a - size_a)
short_b = max(0, min_b - size_b)

over_a  = max(0, size_a - max_a)   if max_a finite else 0
over_b  = max(0, size_b - max_b)   if max_b finite else 0

pref = abs(a * (wa + wb) - E * wa)
```

Then define the score tuple:

```text
score(a) = (
    short_penalty(short_a, short_b, sa, sb, policy.shortage_mode),
    overflow_penalty(over_a, over_b, policy.overflow_mode),
    pref,
    tie_break(a, policy.tie_break),
)
```

Choose the `a` with lexicographically minimal score.

### Default v1 policy

```rust
pub struct SolverPolicy {
    pub shortage_mode: ShortageMode, // default: ByShrinkPriority
    pub overflow_mode: OverflowMode, // default: Uniform
    pub tie_break: TieBreakMode,     // default: PreferA
}
```

Default meaning:

- shortage is distributed away from high-`shrink` leaves
- overflow beyond maxima is uniform-cost
- leftover unit ties go to `A` (left/top)

Suggested enums:

```rust
pub enum ShortageMode {
    Equal,
    ByShrinkPriority,
}

pub enum OverflowMode {
    Uniform,
}

pub enum TieBreakMode {
    PreferA,
    PreferB,
}
```

### Why this model is good

It is unified:

- feasible case: first two penalty components are zero, so it reduces to nearest-to-ideal weighted split
- infeasible shortage: weighted shortage decides who gets hurt
- infeasible overflow: preference decides where excess goes after maxes are inevitably violated

It is deterministic:

- exact integer search space
- exact lexicographic score
- exact tie-break

It is testable:

- a slow exhaustive oracle can brute-force `0..=E` for small sizes and compare exactly

## 7.3 Efficient implementation

The **spec** is exhaustive minimization.

The **implementation** does not need to scan all `0..=E` in production.

Because the score is piecewise linear, the minimum occurs at a small candidate set near:

- `0`
- `E`
- `min_a`
- `E - min_b`
- `max_a` if finite
- `E - max_b` if finite
- `floor(ideal)`
- `ceil(ideal)`

Production code can evaluate just those breakpoints.

But the test suite should include a slow reference allocator that brute-forces every `a` for small bounds.

## 7.4 Recursive solve

Pseudo-code:

```rust
fn solve_node<T>(
    tree: &Tree<T>,
    id: NodeId,
    rect: Rect,
    summaries: &HashMap<NodeId, Summary>,
    policy: &SolverPolicy,
    out: &mut Snapshot,
) {
    out.node_rects.insert(id, rect);

    match &tree.nodes[&id] {
        Node::Leaf(leaf) => {
            record_leaf_violations(id, rect, &leaf.meta, out);
        }
        Node::Split(split) => {
            let sum_a = &summaries[&split.a];
            let sum_b = &summaries[&split.b];

            let total = match split.axis {
                Axis::X => rect.w,
                Axis::Y => rect.h,
            };

            let lead_extent = choose_extent(
                total,
                axis_limits(sum_a, split.axis),
                axis_limits(sum_b, split.axis),
                split.weights,
                sum_a.shrink_cost,
                sum_b.shrink_cost,
                policy,
            );

            let (rect_a, rect_b) = rect_split(rect, split.axis, lead_extent);

            out.split_traces.push(...);

            solve_node(tree, split.a, rect_a, summaries, policy, out);
            solve_node(tree, split.b, rect_b, summaries, policy, out);
        }
    }
}
```

## 7.5 Snapshot

```rust
pub struct Snapshot {
    pub revision: u64,
    pub root: Rect,
    pub node_rects: HashMap<NodeId, Rect>,
    pub split_traces: Vec<SplitTrace>,
    pub violations: Vec<Violation>,
    pub strict_feasible: bool,
}
```

Recommend storing rectangles for **all** nodes, not just leaves. That makes navigation, resize, and debugging much easier.

# 9. State machine model

The state layer owns focus and selection.

```rust
pub struct Session<T> {
    pub tree: Tree<T>,
    pub focus: Option<NodeId>,     // must be a leaf when Some
    pub selection: Option<NodeId>, // leaf or split
    pub revision: u64,
}
```

## 9.1 State invariants

If the tree is empty:

- `focus == None`
- `selection == None`

Otherwise:

- `focus.is_some()`
- `focus` refers to a leaf
- `selection.is_some()`

And:

- if `selection` is a leaf, it must equal `focus`
- if `selection` is a split, it must contain `focus` in its subtree

That is a very good invariant set. It keeps the model crisp and avoids selection/focus drift.

## 9.2 Why geometry-dependent commands require a Snapshot

Directional navigation and directional resize are defined over **solved rectangles**, not topology alone.

So the state layer should not silently use stale geometry.

Use this pattern:

```rust
pub fn focus_dir(&mut self, dir: Direction, snap: &Snapshot) -> Result<(), NavError>;
pub fn grow_focus(&mut self, dir: Direction, amt: u32, strategy: ResizeStrategy, snap: &Snapshot) -> Result<(), OpError>;
```

Every geometry-dependent method must check:

- `snap.revision == self.revision`

Otherwise return `StaleSnapshot`.

That is a very strong design choice.

# 10. Geometric navigation

Use current leaf rectangles from a `Snapshot`.

Given current focused leaf `C`, candidate leaf `R` is eligible in direction `D` if it lies in that half-plane:

- Left: `R.right() <= C.left()`
- Right: `R.left() >= C.right()`
- Up: `R.bottom() <= C.top()`
- Down: `R.top() >= C.bottom()`

For each eligible candidate compute:

- `primary_gap`: face-to-face distance in the requested direction
- `orth_gap`: gap between orthogonal intervals, zero if they overlap
- `orth_center_delta`: absolute center distance on the orthogonal axis
- `tree_order_rank`: deterministic final tie-break

Choose minimal lexicographically:

```text
(primary_gap, orth_gap, orth_center_delta, tree_order_rank)
```

This gives intuitive results:

- nearest pane in that direction first
- aligned panes preferred over diagonally displaced panes
- ties deterministic

This should be a reference-scored function, not a bag of heuristics.

# 11. Resize semantics

This needs to be exact, because sloppy resize semantics create terrible edge bugs.

## 11.1 Important rule

A split is eligible to resize a focused leaf edge in direction `D` **only if that split’s divider lies on that exact solved edge of the focused leaf**.

This is critical.

Merely sharing the same axis is not enough.

Example:

- leaf is somewhere inside the left subtree
- root split is `Axis::X`
- but another sibling inside that left subtree occupies the leaf’s right side
- moving the root divider does **not** move that leaf’s right edge

So eligibility must be based on actual solved geometry.

## 11.2 Eligible split test

For focused leaf rect `F` and ancestor split `S`:

For `Direction::Right`:

- `S.axis == Axis::X`
- `F.right() == rect_of(S.a).right()`
- focus lies in subtree `S.a`

For `Direction::Left`:

- `S.axis == Axis::X`
- `F.left() == rect_of(S.b).left()`
- focus lies in subtree `S.b`

For `Direction::Down`:

- `S.axis == Axis::Y`
- `F.bottom() == rect_of(S.a).bottom()`
- focus lies in subtree `S.a`

For `Direction::Up`:

- `S.axis == Axis::Y`
- `F.top() == rect_of(S.b).top()`
- focus lies in subtree `S.b`

The eligible splits on the path to root are ordered nearest-first.

## 11.3 Divider motion at a split

Each eligible split has a current divider position expressed by `a_current`.

Under current strict bounds, the split has a feasible interval for `a`:

- `lo <= a <= hi`

You can derive `lo` and `hi` from subtree mins/maxes under the current cross-size.

Then available strict slack is:

- increase-`a` slack: `hi - a_current`
- decrease-`a` slack: `a_current - lo`

Mapping outward leaf-edge motion to divider motion:

- Right / Down outward growth means `a += delta`
- Left / Up outward growth means `a -= delta`

## 11.4 Resize strategies

```rust
pub enum ResizeStrategy {
    Local,
    AncestorChain,
    DistributedBySlack,
}
```

### Local

Use the nearest eligible split only.
Clamp movement to that split’s strict slack.

### AncestorChain

Walk eligible splits nearest to farthest.
Consume as much requested movement as possible at each split.
Stop when request is satisfied or no more strict slack exists.

This gives the “borrow outward” feel editors often want.

### DistributedBySlack

Compute total strict slack across all eligible splits.
Clamp request to that total.
Distribute requested movement proportionally to each split’s available slack.

This minimizes over-distorting the nearest boundary.

## 11.5 How weights are updated

Resize does not store pixels. It rewrites split preferences.

For each touched split:

- compute new desired `a_new`
- set split weights proportional to `(a_new, total - a_new)`
- canonicalize by GCD
- preserve zero if one side is fully collapsed in preference
- reject `(0, 0)`

Example:

```rust
fn canonicalize_weights(a: u32, b: u32) -> WeightPair
```

If both are nonzero, divide by `gcd(a, b)`.

If one side is zero, keep `(0, 1)` or `(1, 0)`.

# 12. Structural operations

I would make the mutation surface explicit and small.

## 12.1 Essential operations

### Insert root

Create the first leaf if the tree is empty.

### Split focused leaf

Replace the focused leaf with a split whose children are:

- old leaf
- new leaf

```rust
split_focus(axis, slot, payload, meta, initial_weights)
```

### Wrap selection

Wrap any selected node with a new sibling leaf under a new split.

```rust
wrap_selection(axis, slot, payload, meta, initial_weights)
```

### Remove focused leaf

Delete the focused leaf.
Collapse any resulting unary parent.

### Swap disjoint nodes

Swap two nodes or subtrees, disallowing ancestor/descendant pairs.

### Move selected subtree as sibling of target

Detach selected subtree.
Attach it next to a target node under a new split.
Disallow moving into own descendant.

### Toggle split axis

On a selected split only.

### Mirror subtree

Recursively swap children of all splits in the subtree whose axis matches the mirror axis.

- `mirror(Axis::X)` => left/right mirror
- `mirror(Axis::Y)` => top/bottom mirror

### Rebalance weights

Keep structure, rewrite weights.

### Apply preset

Rebuild selected subtree from its leaves in stable leaf order.

## 12.2 Repair rules after mutation

After any mutation:

1. If focused leaf survives, keep it.
2. Otherwise, choose the first leaf of the structural replacement subtree at the mutation site.
3. If tree became empty, `focus = None`.

For selection:

1. If selected node survives and contains focus, keep it.
2. Else `selection = focus`.

These rules are simple and deterministic.

# 13. Presets

Presets should be **subtree rebuilds over an ordered leaf list**, not special runtime node types.

That keeps the solver uniform.

Use stable leaf order:

- DFS leaf order, visiting child `A` before child `B`

Preserve:

- leaf IDs
- payloads
- leaf metadata

Split IDs may be recreated.

## 13.1 Balanced

Build a near-balanced binary tree by recursively splitting the ordered leaf slice at midpoint.

```rust
Balanced { start_axis: Axis, alternate: bool }
```

Suggested builder:

- split midpoint with left bias on odd counts
- alternate axis by depth
- set split weights by descendant leaf counts

This gives a structurally balanced tree with roughly equal leaf share.

## 13.2 Dwindle

Build a Fibonacci/dwindle chain:

```text
L0
-> Split(start_axis, L0, rest)
-> rest = Split(toggled_axis, L1, rest)
-> ...
```

Equivalent recursive shape:

```text
Dwindle([L0, L1, L2, L3], X)
= Split(X, L0, Split(Y, L1, Split(X, L2, L3)))
```

Use equal `1:1` weights at each split.

Config:

```rust
Dwindle {
    start_axis: Axis,
    new_leaf_slot: Slot, // usually B
}
```

## 13.3 Tall

One master leaf on one side, stack of remaining leaves on the other.

```rust
Tall {
    master_slot: Slot,           // usually A
    root_weights: WeightPair,    // default 1:1
}
```

Build:

- root split on `Axis::X`
- master = first leaf
- stack = linear equal builder on `Axis::Y`

The stack builder should use weights `(1, remaining_count - 1)` recursively so stack leaves are equal-height.

## 13.4 Wide

Same as Tall, rotated:

- root split on `Axis::Y`
- stack axis `Axis::X`

## 13.5 Why presets are rebuilds

Because that makes them:

- deterministic
- easy to test
- independent from current bounds
- orthogonal to solving

# 14. Rebalance modes

This is separate from presets.

Rebalance keeps structure and only changes weights.

```rust
pub enum RebalanceMode {
    BinaryEqual,   // every split => (1,1)
    LeafCount,     // split weights => descendant leaf counts
}
```

`BinaryEqual` is a local “make every divider 50/50”.

`LeafCount` is stronger. It makes recursive chains produce equal leaf share where constraints allow.

    pub fn apply_preset(&mut self, preset: PresetKind) -> Result<(), OpError>;
}

That is a solid v1 ergonomic surface.

# 16. Proof obligations and executable “can’t fake it” tests

This is where you can make the implementation genuinely hard to cheat.

Do not rely on ordinary unit tests alone.

Use four layers:

- exact small-state oracles
- structural invariants
- metamorphic tests
- random command traces

## 16.1 Reference allocator oracle

In test code only, implement:

```rust
fn choose_extent_oracle(spec: PairSpec) -> u32 {
    (0..=spec.total)
        .min_by_key(|&a| exhaustive_score(spec, a))
        .unwrap()
}
```

For small extents, brute-force every possible `a`.

Then assert that the production allocator returns the exact same result for:

- random weights
- random mins/maxes
- random shortage priorities
- every total extent up to some small cap

This is a brutal test. A heuristic solver will not survive it.

## 16.2 Full small-tree reference solver

For small trees and small bounds, solve using:

- production solver
- reference solver using brute-force allocation at every split

Compare:

- every node rect
- every leaf rect
- every violation
- `strict_feasible`

Run this over:

- all binary tree shapes up to 5 leaves
- randomized limits
- bounds up to maybe `8x8` or `10x10`

That is the single best anti-cheat test.

## 16.3 Raster partition proof

For small bounds, rasterize the root into cells.

For each leaf rect:

- write its leaf ID into every covered cell

Assert:

- every cell inside root written exactly once
- no overlap writes
- no unwritten cells

This proves exact partition, not just area conservation.

## 16.4 Summary correctness oracle

For each small random subtree, brute-force strict feasibility over all bounds up to a small cap and verify:

- summary `min_w/min_h` are the minimal feasible extents
- summary maxima match the maximal extents where strict feasibility still holds, when finite

This directly validates the bottom-up summary equations.

## 16.5 Determinism tests

For a fixed tree and bounds:

- solving twice gives byte-identical snapshot output
- serializing and deserializing the tree then solving gives identical output
- replaying a command log gives identical output

## 16.6 Metamorphic tests

These are excellent for catching accidental asymmetry.

### Mirror symmetry

For a tree `T` and root `R`:

- solve `T`
- build `mirror_x(T)`
- solve mirrored tree
- mirror original rects across `R`
- assert equality

Do the same for `Y`.

### Scale symmetry

Choose factor `k`.
Scale:

- bounds
- mins
- maxes

Solve both.
Assert scaled output is exactly `k * original`.

This is very strong when arithmetic is exact integer.

### Swap involution

Swap two disjoint nodes twice.
Tree and layout must return exactly.

### Split-remove roundtrip

Split a leaf with a new sibling, then immediately remove the new sibling.
Original tree and layout must return exactly, aside from revision and removed split ID.

### Rebalance idempotence

Applying the same rebalance twice must be a no-op the second time.

### Preset idempotence

Applying the same preset twice to an already canonical subtree should not change layout or structure.

## 16.7 Navigation oracle

Implement a tiny brute-force nav scorer from the spec.

For every leaf and direction in small solved layouts:

- production `focus_dir` candidate must equal oracle candidate

This catches “looks okay manually” bugs.

## 16.8 Resize maximality tests

For a focused leaf edge and strategy:

- compute theoretical strict slack from eligible splits
- request more than available
- assert actual motion equals exactly the maximum possible motion, no more and no less

For `AncestorChain`, assert:

- achieved movement = min(requested, sum of path slack)

For `Local`, assert:

- only nearest eligible split changes

For `DistributedBySlack`, assert:

- touched deltas sum to requested clamped movement
- individual deltas are proportional to available slack, modulo deterministic remainder assignment

## 16.9 Random command traces

Use `proptest` or `bolero` to generate sequences of commands:

- split
- remove
- wrap
- swap
- move
- rebalance
- preset
- focus navigation
- resize

After every command:

- validate tree invariants
- solve and validate partition invariants
- on small bounds, compare to reference solver

This is where a lazy implementation usually collapses.

## 16.10 Hidden CI seeds

For CI:

- keep public deterministic seeds in repo
- also run a hidden larger random seed batch in CI

That keeps people from hardcoding only the visible cases.

# 17. Useful proof sketches

These are worth writing into docs.

## 17.1 Partition theorem

**Claim:** solving any tree returns rectangles whose leaves exactly partition the root.

**Sketch:**
Base case: a leaf receives exactly its parent rect.
Inductive step: a split divides its parent rect into two disjoint half-open child rects whose union is the parent. By induction, each child subtree partitions its child rect. Therefore the union of all leaf rects is the root rect and overlaps are impossible.

## 17.2 Determinism theorem

**Claim:** solve is deterministic.

**Sketch:**
The tree, bounds, and policy are immutable inputs. Every split allocation is chosen by lexicographically minimizing a pure score with a deterministic tie-break. Recursion order is fixed. Therefore output is unique.

## 17.3 Summary exactness theorem

**Claim:** subtree summaries are exact strict feasibility envelopes in v1.

**Sketch:**
Because v1 constraints are separable by axis and there are no aspect-ratio couplings, the minimum required extent of a split subtree on its split axis is additive, and on the cross-axis is the maximum of child minima. The same dual rule holds for maxima. Induct over the tree.

## 17.4 Resize maximality theorem

**Claim:** `AncestorChain` growth moves a focused edge by exactly the total strict slack available along eligible boundaries, clamped by the request.

**Sketch:**
Only eligible boundaries can move that leaf edge. Their effects add monotonically in the requested direction. Each split is locally clamped by strict feasibility. Therefore the path sum is both achievable and maximal.

# 18. One implementation detail that matters a lot

Use `u128` internally for score arithmetic.

The term:

```text
abs(a * (wa + wb) - E * wa)
```

and penalty products can overflow `u32` or `u64` surprisingly quickly under fuzzing.

Externally, keep rectangles in `u32/i32`.
Internally, score math should be wider.

let mut sess = Session::new();

let a = sess.insert_root(BufferId(1), LeafMeta::default())?;

# 22. Final recommendation

I would ship v1 around these commitments:

- single-root binary split tree
- half-open integer rects
- leaf-authored hard min/max constraints
- separate weights and priorities
- exact bottom-up summaries
- deterministic best-effort solver with strict certification
- snapshot-gated geometry commands
- edge-based resize eligibility
